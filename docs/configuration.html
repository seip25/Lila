Configuration & ENV_CONFIG

Lila uses a .env file as the single source of truth for all application settings. Framework variables like DEBUG, PORT, and SECRET_KEY are loaded automatically with proper type casting (string, integer, boolean). For any custom variable you add to .env, the ENV_CONFIG proxy gives you instant access from anywhere in your project — no extra registration needed.

The .env File

After running lila-init, your project includes a .env file with all framework defaults. To change any setting, just edit this file — no code changes are needed.

.env 

# ─── Server ────────────────────────────────────────────
SECRET_KEY='your-secret-key-here'
DEBUG=True
PORT=8001
HOST=127.0.0.1
JIT=False
WORKERS=1
MINIFY_HTML=False

# ─── Application URL ──────────────────────────────────
# Production URL for sitemaps, robots.txt, canonical links.
# Leave empty in development — uses http://HOST:PORT automatically.
APP_URL=""

# ─── Project Metadata ─────────────────────────────────
TITLE_PROJECT='My App'
VERSION_PROJECT='1'
DESCRIPTION_PROJECT='My awesome project'
LANG_DEFAULT='en'

# ─── SEO Defaults ─────────────────────────────────────
DESCRIPTION_DEFAULT='A Python web framework'
KEYWORDS_DEFAULT='Python, web, framework'
AUTHOR_DEFAULT='Your Name'

# ─── Your Custom Variables ─────────────────────────────
# Add any variable here and access it via ENV_CONFIG
STRIPE_API_KEY='sk_test_...'
REDIS_URL='redis://localhost:6379'

Built-in Framework Variables

These variables are recognized by the framework automatically. They are type-casted (string → int, bool, etc.) and available as module-level constants via from app.config import ...

Variable Type Default Description
SECRET_KEY str "" Secret key for signing sessions and tokens
PORT int 8001 Server port
HOST str "127.0.0.1" Server host address
APP_URL str "" Production URL (sitemaps, robots.txt, canonical links). Falls back to http://HOST:PORT when empty.
DEBUG bool True Enable debug mode (hot reload, error pages)
JIT bool True Enable Python JIT compilation
WORKERS int 1 Number of Uvicorn worker processes
MINIFY_HTML bool False Minify HTML output in templates
TITLE_PROJECT str "Lila project" Project title (used in templates and SEO)
VERSION_PROJECT str "1" Project version
DESCRIPTION_PROJECT str "" Project description for OpenAPI docs
LANG_DEFAULT str "en" Default language for i18n
DESCRIPTION_DEFAULT str "A Python web framework" Default meta description for SEO
KEYWORDS_DEFAULT str "Python, web, framework" Default meta keywords for SEO
AUTHOR_DEFAULT str "Seip" Default author for meta tags

Using Framework Variables

All built-in variables are exported as module-level constants from app.config. Import them directly — they are loaded once at startup with automatic type casting.

Importing framework variables 

from app.config import DEBUG, PORT, HOST, SECRET_KEY

if DEBUG:
    print(f"Running in debug mode on {HOST}:{PORT}")

ENV_CONFIG — Access Any .env Variable

ENV_CONFIG is a dict-like proxy that gives you instant access to any variable defined in your .env file — including custom ones that are not part of the framework. No registration, no configuration — just add the variable to .env and read it.

Using ENV_CONFIG 

from app.config import ENV_CONFIG

# Access with [] — raises KeyError if not found
stripe_key = ENV_CONFIG["STRIPE_API_KEY"]

# Access with .get() — returns a default if not found
redis_url = ENV_CONFIG.get("REDIS_URL", "redis://localhost:6379")

# Check if a variable exists
if "CUSTOM_FEATURE_FLAG" in ENV_CONFIG:
    print("Feature flag is set!")

# Works with any variable in .env — no registration needed
api_version = ENV_CONFIG.get("API_VERSION", "v1")

Real-World Example

A common pattern is combining framework variables with custom ENV_CONFIG variables in your routes and services:

app/routes/api.py 

from lila.core.routing import Router
from lila.core.request import Request
from lila.core.responses import JSONResponse
from app.config import DEBUG, ENV_CONFIG

router = Router("api")

@router.get("/status")
async def status(request: Request):
    return JSONResponse({
        "debug": DEBUG,
        "api_version": ENV_CONFIG.get("API_VERSION", "v1"),
        "feature_x": ENV_CONFIG.get("ENABLE_FEATURE_X", "false"),
    })

@router.post("/payment")
async def create_payment(request: Request):
    # Access your Stripe key — defined only in .env
    stripe_key = ENV_CONFIG["STRIPE_API_KEY"]
    # ... payment logic
    return JSONResponse({"success": True})

How app/config.py Works

After running lila-init, your app/config.py is clean and minimal. The ConfigLoader handles all the .env loading, type casting, and production caching internally:

app/config.py 

import os
from os import path
from lila.core.config import ConfigLoader, ENV_CONFIG

# All framework variables loaded and exported automatically
_cfg = ConfigLoader.load(cache_dir=path.dirname(__file__))
globals().update(_cfg)

# Project-specific paths (not loaded from .env)
PATH_LOG_BASE_DIR = "app/logs"
PATH_TEMPLATE_NOT_FOUND = "lila/404"
PATH_TEMPLATES_HTML = "resources/html/"
PATH_TEMPLATES_MARKDOWN = "resources/markdown/"
PATH_LOCALES = "app/locales"
PATH_UPLOADS = path.join(os.getcwd(), "public", "img", "uploads")

That's it — no duplication. Each framework variable is defined once in the schema and exported automatically. To change any value, edit .env and restart the server.

Production Caching

When DEBUG=False, Lila automatically generates a config_cache.py file in your app/ directory. This compiled cache is loaded on subsequent startups without re-reading .env, giving you faster boot times in production.

The cache is automatically invalidated and regenerated whenever the .env file is modified (checked via file modification timestamp).

⚡

Performance Note

ENV_CONFIG reads values that were loaded at startup — it does not re-read .env on every access. This means there is zero performance overhead when using ENV_CONFIG in your route handlers.

API Reference

Method Returns Description
ENV_CONFIG["KEY"] str Get a value. Raises KeyError if not found.
ENV_CONFIG.get("KEY", default) str | None Get a value with a fallback default. Returns None if not found and no default given.
"KEY" in ENV_CONFIG bool Check if a variable exists in the configuration.
ConfigLoader.load(cache_dir) dict Load all framework config from .env. Called once in app/config.py.
ConfigLoader.get("KEY", default) any Get any config value. Checks framework fields first, then raw .env values.
